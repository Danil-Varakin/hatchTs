# Hatch

**Структурные семантические патчи для исходного кода.** Вместо хрупких номеров
строк патч Hatch описывает, *где* менять код, в терминах самой структуры кода — а
инструмент находит место. Порт оригинального Python-прототипа на TypeScript:
типизированный AST и npm-нативный пайплайн (без Python в сборке).

## Зачем

Патчить быстро меняющийся upstream классическими `.patch`-файлами
больно: вставили пару строк выше — и номера строк во всех ханках поехали. Hatch
описывает изменение декларативно — «вставить вот это *после того include*, *перед
namespace*» — тремя операторами. Позиция ищется по разобранной структуре файла,
поэтому переформатирование и несвязанные правки upstream патч не ломают.

Две команды:

- **`apply`** — применить `.md`-инструкции к исходному файлу.
- **`generate`** — сравнить две версии файла и породить `.md`-инструкции.

`generate`, затем `apply` замыкают round-trip: применение порождённого патча к
старому файлу воспроизводит новый.

## Язык: три оператора

Патч — это Markdown с парами блоков `match`/`patch`. Блок `match` пишется на
целевом языке, операторы вставлены между строками:

| Оператор | Значение |
|----------|----------|
| `...`    | пропуск балансированного фрагмента до следующего якоря (все вхождения, с откатами) |
| `>>>`    | точка вставки |
| `<<<`    | конец диапазона замены (от `>>>`) |
| `^..`    | пропуск до **первого** вхождения якоря |
| `..^`    | пропуск до **последнего** вхождения |
| `^n..`   | пропуск до **n-го** вхождения (1-based) |

Оператор распознаётся только как отдельное *слово* (пробел или край строки с
обеих сторон), поэтому `template <typename... Args>` остаётся литералом. Настоящее
` ... ` в коде экранируется как ` \... `.

## Пример

Вставить `#include "base/feature_override.h"` сразу после существующего include,
перед стандартными библиотечными includes (`content/common/features.cc`):

````markdown
### match
```cpp
...
// found in the LICENSE file.
 #include "content/common/features.h"

 >>>
#include "base/feature_list.h"

 ...
```
### patch
```cpp
#include "base/feature_override.h"
#include "build/build_config.h"
```
````

Читается так: *пропусти что угодно, найди строку про LICENSE, сразу за ней include
`features.h`, **вставь здесь**, следом обязан идти include `feature_list.h`, затем
что угодно до конца файла.* Больше разобранных примеров (вложенные namespace,
вставка метода после закрывающей скобки другого метода) — в тестовых фикстурах.

## Использование

```bash
# apply
hatch apply --match changes.md --in src/main.cpp --out src/main.cpp

# generate
hatch generate --in new.cpp --in-old old.cpp --out changes.md
# ...или сравнить с git-веткой
hatch generate --in src/main.cpp --branch master --out changes.md
```

### Опции `apply`
```
--match <файл>     путь к .md с блоками match/patch
--patch <файл>     отдельный файл патча (опц.)
--in <файл>        входной исходник
--out <файл>       выходной файл
--language <язык>  cpp | python (иначе по расширению)
--dry-run          показать правки, ничего не писать
--verify           только проверить применимость; код выхода для CI
```

### Опции `generate`
```
--in <файл>        новая версия файла
--in-old <файл>    старая версия
--branch <ветка>   git-ветка для сравнения (по умолчанию: master)
--language <язык>  cpp | python (иначе по расширению)
-a, --agreement    подтверждать каждое совпадение интерактивно
```

### Коды выхода (для CI)
`0` успех · `2` ошибка разбора · `3` нет совпадения (показывает «самую глубокую»
точку отказа) · `4` неоднозначно (показывает конкурирующие позиции) · `5` уже
применено (идемпотентность) · `1` непредвиденное.

## Три правила, зафиксированные волевым актом (из синтаксиса не выводятся)

Они осознанные и стабильные; патчи на них опираются:

1. **`^n..` — 1-based.** `^1..` ≡ `^..`.
2. **`<<<` заменяет ВКЛЮЧИТЕЛЬНО.** Литералы между `>>>` и `<<<` — это «старый
   код»: обязаны совпасть, но в выход не идут — их место занимает тело патча.
   Литералы *снаружи* меток — контекст, сохраняются.
3. **Закрытие области зависит от языка.** В C++ область закрывается литералом
   `}`. В Python закрывающего токена нет — конец области выражается якорем-литералом
   на внешнем отступе либо неявно через `...` до строки меньшего отступа.

## Статус

Это порт в процессе. Готово и протестировано: язык-нейтральное ядро парсинга
(`src/core/` — AST, ошибки, single-pass парсер, принтер) и канонизатор литералов
C++ (`src/lang/cpp/normalize.ts`), всё зелёное под строгим tsconfig. В работе:
C++ `SourceMap` (на tree-sitter + общий канонический перенос координат),
матчер/патчер, конвейер `generate` и Python-адаптер. Постатусно по модулям — в [docs/structure.md](./docs/structure.md),
план — в [docs/phase-*.md](./docs).

## Сборка и запуск

Исходники — `.ts`, исполняются напрямую на **Node 22+** через strip-types, без
шага сборки. TypeScript используется только для проверки типов.

```bash
node --experimental-strip-types src/cli/index.ts apply --match ... --in ...
npx tsc -p tsconfig.json   # только проверка типов (noEmit)
```

Разбор структуры — через **tree-sitter** (`web-tree-sitter`, WASM-грамматики:
кросс-платформенно, без нативной сборки); грузятся разово на старте. Рабочий
процесс и обоснование строгого tsconfig — в [CONTRIBUTING.ru.md](./CONTRIBUTING.ru.md).
