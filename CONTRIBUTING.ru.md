# Как участвовать

> English version: [CONTRIBUTING.md](./CONTRIBUTING.md)

Спасибо, что помогаете строить Hatch. Этот файл — практическое «как над этим
работать»; обоснование дизайна — в [ARCHITECTURE.md](./ARCHITECTURE.md) (англ.).

## Требования

- **Node 22+** (исходники исполняются через strip-types; шага сборки нет).
- Для ядра больше ничего. С фазы 2 адаптеры языков зависят от **`web-tree-sitter`**
  и файлов грамматик `.wasm` (`tree-sitter-cpp.wasm`, позже `tree-sitter-python.wasm`).
  Это WASM, не нативщина — кросс-платформенно, без компилятора и `node-gyp`. Грузятся
  асинхронно: адаптер даёт `init(): Promise<void>`, его ждут один раз на старте CLI,
  после чего `buildMap` синхронный. `.wasm` класть в `files` package.json.
- CLI-слою позже понадобится `@types/node` (см. закомментированный блок в
  `tsconfig.json`); `web-tree-sitter` тянет свои типы.

## Запуск и проверка типов

Запустить любой `.ts` напрямую:

```bash
node --experimental-strip-types src/cli/index.ts apply --match changes.md --in src/main.cpp
```

Проверить типы всего дерева исходников (JS не собирается — `noEmit`):

```bash
npx tsc -p tsconfig.json
```

Зелёный `tsc` обязателен перед каждым коммитом. Ядро уже проходит с нулём ошибок
под строгим конфигом ниже.

## Строгий tsconfig (без причины не ослаблять)

Конфиг осознанно строгий, каждый флаг оправдан:

- **`noEmit` + `allowImportingTsExtensions`** — tsc только проверяет типы;
  исходники исполняются через `node --experimental-strip-types`, поэтому импорты
  пишутся с расширением `.ts`. Если когда-нибудь перейдёшь на сборку в `dist/` —
  вернуть опции эмита и добавить `rewriteRelativeImportExtensions` (перепишет
  `.ts`→`.js`).
- **`verbatimModuleSyntax` + `isolatedModules`** — type-only импорты ОБЯЗАНЫ быть
  `import type { … }`. Весь `core/` так и написан (например, `parser.ts` импортит
  типы из `ast.ts` через `import type`, а значение `ParseError` — обычным `import`).
- **`noUncheckedIndexedAccess`** — индексный доступ (`arr[i]`, `m.groups[k]`) даёт
  `T | undefined`. Обрабатывать явно (`?? ''`, `.entries()`, проверки).
- **`exactOptionalPropertyTypes`** — `hint?: string` НЕ включает `undefined`;
  присваивать поле только когда значение есть (`if (hint !== undefined) …`).

## Соглашения по коду

- **Никаких parameter properties в конструкторах** — strip-types их не держит.
  Объявлять поля явно (см. `errors.ts`).
- Держать `core/` язык-нейтральным. Если правка «для C++» или «для Python» — ей
  место в адаптере под `lang/`. Это и есть тест на границу.
- Атомарная запись (`infra/fs.ts`): temp-файл + `rename`. Инструмент правит
  исходный код; оборванный файл = сломанная сборка.
- Ошибки наследуют `HatchError` и несут `exitCode`.

## Как организован код

Полное дерево и постатусно по модулям — в [docs/structure.md](./docs/structure.md).
Кратко:

- `src/core/` — семантика Hatch, без знания о языке: `ast.ts`, `errors.ts`,
  `parser.ts` (один проход, инкрементальная валидация), `printer.ts`, плюс
  будущие `matcher.ts` и `patcher.ts`.
- `src/lang/` — вся балансировка и нормализация: `source-map.ts` (центральный
  контракт), папки по языкам (`cpp/`, `python/`), у каждой свой `normalize.ts` и
  строитель карты.
- `src/generate/`, `src/infra/`, `src/cli/` — конвейер `generate`, побочные
  эффекты (git, атомарная fs) и CLI на commander.

## План фаз

Разработка разбита так, чтобы каждая фаза давала тестируемый артефакт, а самый
рискованный модуль (карта структуры) получил фундамент раньше матчера. Подробно
по фазам — в [docs/phase-1..5-*.md](./docs).

1. **Ядро парсинга (без языка).**  готово и протестировано — `ast`, `errors`,
   `parser`, `printer`. Round-trip зелёный, запрещённые комбинации отвергаются с
   номерами строк.
2. **C++ `SourceMap` (tree-sitter).** `source-map.ts` (контракт),
   `cpp/normalize.ts`, затем `cpp/treesitter.ts` (обёртка web-tree-sitter),
   общий `lang/canon.ts` (канон + выравнивание непробельных для `toOriginal` и
   переноса смещений узлов) и `cpp/index.ts`. Пары скобок даёт tree-sitter; канон/
   нормализация — наши. Карта тестируется в отрыве.
3. **Матчер + патчер (вертикальный срез `apply`).** Обход курсором по шагам в
   КАНОНИЧЕСКИХ координатах, окна через `pair`, метки в оригинал через `toOriginal`.
   Ханки применяются ПОСЛЕДОВАТЕЛЬНО (каждый против текущего состояния), одна
   атомарная запись в конце. Первый сквозной патч.
4. **`generate`.** diff → synth (подъём по `enclosing()` той же карты) → printer.
   Последовательное применение позволяет synth выпускать зависимые ханки на кучных
   правках. Замыкает round-trip системы.
5. **Python + укрепление.** `python/normalize.ts` (наш) поверх структуры от
   `tree-sitter-python`, режим подтверждения `-a`, канон-с-учётом-строк через узлы
   строк tree-sitter, устойчивость к `ERROR`-узлам.

## Тесты

- `test/unit/` — parser, source-map (ОТДЕЛЬНО от матчера!), cpp/normalize,
  py/normalize, matcher.
- `test/roundtrip/` — `parse(print(ast)) ≡ ast`.
- `test/golden/` — четыре реальных патча с ожидаемым результатом.

Инварианты из [ARCHITECTURE.md](./ARCHITECTURE.md) держать зелёными на каждом
коммите.

## Два решения зафиксировать в README до появления реальных патчей

Из синтаксиса не выводятся, объявить волевым актом, иначе патчи разойдутся в
трактовках. Уже в README — держать авторитетными: `^n..` 1-based; `<<<` заменяет
включительно; закрытие области — `}` в C++ и якорь на внешнем отступе (или неявно
`...`) в Python.
